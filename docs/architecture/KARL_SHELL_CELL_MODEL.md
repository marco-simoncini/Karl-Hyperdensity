# KARL Shell and Cell Model

**Status:** Architecture foundation (Sprint 1) — contract definition; no CRD or controller implementation in this milestone.

**Related:** `KARL_HOST_RUNTIME_VISION.md`, ADR-0001, ADR-0002.

---

## Definitions

### Shell

A **Shell** is the **primary product-facing abstraction** for a workload experience: what a user, administrator, or upstream catalog selects and reasons about.

Examples (non-exhaustive):

- Windows desktop session (DaaS).
- Single Windows application delivered via RemoteApp-style experience.
- Linux workspace / “environment.”
- Stateful app or service with a defined SLA class.
- VM-like workload experience (legacy or native).
- Migration-era representation of an existing VM.

Shells are **stable identifiers** in APIs, RBAC, billing, and policy. They are **not** tied to a single low-level Kubernetes object.

### Cell

A **Cell** is the **node-materialized execution unit** that KHR (or, during transition, legacy controllers) creates, mutates, or destroys to realize all or part of a Shell.

Examples:

- Kubernetes Pod / container(s).
- systemd-managed workload slice.
- Windows session + job objects + profile mounts (via Windows Host Agent).
- Windows App Cell (RemoteApp-style surface).
- Native VM-like Cell (future provider; see FluidVirt discussion in vision doc).
- **KubeVirt VirtualMachineInstance** (legacy provider).
- Cloud-backed worker instance (adaptive provider).

### Difference (one paragraph)

**Shell** answers “what experience or workload contract does the customer have?” **Cell** answers “what concrete runtime object exists on a specific host right now to back that contract?” One Shell may map to one Cell, or temporarily to **zero** Cells (draining), or to **multiple** Cells during controlled migration / canary / multi-region patterns—always through explicit product semantics, not accidental leakage of internal objects.

## Supporting primitives

### ShellClass

Typed behavior and capability profile for a Shell: supported RuntimeProviders, default ResourcePort templates, compliance class (e.g., Linux live envelope vs Windows session envelope), and UX hints for Dashboard / KARL-APP.

### ShellPool

Schedulable inventory and placement domain for Shells (Windows pool, Linux pool, GPU pool, “legacy KubeVirt class,” cloud pool). Pools export **capacity signals** to Hyperdensity and Grande Padre.

### ShellLease

Time-bounded **commercial and operational reservation** binding identity, pool, SLA, cost policy, and gateway rights (e.g., RDP/RemoteApp issuance). ShellLease is the natural handoff object between **catalog / DirectoryService / Licenziatore** and **execution + gateway** layers.

### ShellProfile

Persistent user/environment profile: storage mounts, policy baseline, allowed apps, telemetry tags.

### Host / HostPool

**Host:** registered node with declared hardware, drivers, maintenance state, and KHR version.  
**HostPool:** coarse placement and failure domain for Cells (rack, cluster slice, edge site).

### RuntimeProvider

Plugin implementing materialization for a subset of ShellClasses into Cells. Each provider publishes **ResourcePort** capability matrices (see fusion doc).

### GatewayRoute

Declares how a Shell reaches a user (e.g., via `rdp-GW`, web client, SSM-like broker). Separates **execution** from **ingress**.

### WindowsApp

Catalog-level application object: identity, install evidence, icon, launch parameters, RemoteApp mapping hints. WindowsApp + ShellLease drives **Windows App Shell** behavior.

## Conceptual YAML (illustrative only — not an implemented API)

### Shell (Linux environment)

```yaml
apiVersion: karl.io/v1alpha1
kind: Shell
metadata:
  name: dev-linux-001
spec:
  shellClass: linux.workspace.v1
  shellPool: pool-linux-default
  resourceProfile:
    cpuMillis: 4000
    memoryGi: 16
  runtimePreference:
    primaryProvider: linux.pod.v1
    fallbackProviders: ["linux.systemd.v1"]
```

### Cell (materialized — typically generated by controller/KHR)

```yaml
apiVersion: karl.io/v1alpha1
kind: Cell
metadata:
  name: dev-linux-001-cell-a7f3
  labels:
    karl.io/shell: dev-linux-001
spec:
  shellRef: dev-linux-001
  hostRef: node-42
  runtimeProvider: linux.pod.v1
  resourcePorts:
    cpu: { mode: envelope }
    memory: { mode: envelope }
  providerHandle:
    namespace: workloads
    podName: ws-dev-linux-001
```

### Shell (Windows App)

```yaml
apiVersion: karl.io/v1alpha1
kind: Shell
metadata:
  name: finance-app-calc
spec:
  shellClass: windows.app.v1
  shellPool: pool-windows-finance
  windowsAppRef: msft-excel-addin-claims
  gatewayRoutes:
    - type: rdp.remoteapp.v1
      gatewayClass: karl.rdpgw.v1
```

## Legacy mapping: KubeVirt VM

| Legacy concept | Shell view | Cell view |
|----------------|------------|-----------|
| KubeVirt `VirtualMachine` / `VirtualMachineInstance` | `shellClass: vm.kubevirt.legacy.v1` (or mapped Linux/Windows shell class) | `runtimeProvider: kubevirt.legacy.v1` with provider handle `{vmName, namespace}` |

Dashboard and APIs should **default user language to Shell**, optionally exposing “technical view” for operators.

---

## Mapping: Windows App Shell

- **ShellClass:** `windows.app.v1`
- **Cells:** Windows App Cell on a Windows Host Agent–managed node; may include sidecar Cells for logging only where supported.
- **Gateway:** `rdp-GW` issues RDP file / connection bundle using RemoteApp-related fields already present in the Go RDP model (`remoteapplicationmode`, `remoteapplicationprogram`, etc.).

---

## Mapping: Linux Cell

- **ShellClass:** `linux.workspace.v1` / `linux.container.v1`
- **Cells:** Pod or systemd unit; **ResourcePort** may advertise cgroup-based CPU/RAM envelope modes consistent with current Hyperdensity Linux guarantees documented in `marco-simoncini/Karl-Hyperdensity` (see existing `docs/architecture/hyperdensity-overview.md`).

---

## Mapping: Native VM-like Cell

- **ShellClass:** `vm.native.v1` (future)
- **Cells:** VM-like object via FluidVirt or successor provider when product gates pass.
- **Dependency:** explicit **guest cooperation** and provider capability flags; never implied by Shell name alone.

---

## Governance notes

- **Inventory / DLP / DirectoryService / Licenziatore** attach policy to **Shell** and **ShellLease** identifiers.
- **Karl-Warden** remains relevant for enforcement hooks and host pool integrity as KHR registers truth from the node.

This model is the **lingua franca** for cross-repository work in subsequent sprints.
